Spring循环依赖及其解决方案

Posted Mar 1 2021-03-01T16:25:00+08:00 by Kol Huang

问题描述

在Spring的文档中，这样描述循环依赖问题：

If you use predominantly constructor injection, it is possible to create an unresolvable circular dependency scenario.

For example: Class A requires an instance of class B through constructor injection, and class B requires an instance of class A through constructor injection. If you configure beans for classes A and B to be injected into each other, the Spring IoC container detects this circular reference at runtime, and throws a BeanCurrentlyInCreationException.

One possible solution is to edit the source code of some classes to be configured by setters rather than constructors. Alternatively, avoid constructor injection and use setter injection only. In other words, although it is not recommended, you can configure circular dependencies with setter injection.

Unlike the typical case (with no circular dependencies), a circular dependency between bean A and bean B forces one of the beans to be injected into the other prior to being fully initialized itself (a classic chicken-and-egg scenario).

如果类A依赖于B，类B又依赖于A，那么就形成了循环依赖，这通常是由于基于构造器的依赖注入导致的。在运行时，Spring IoC容器会检测到循环依赖问题，并抛出BeanCurrentlyInCreationException

解决方案

一个可能的解决方案是修改源码，让依赖注入通过Setter方式注入，而不是构造器方式。

如前的例子所述，类A和B相互依赖，双方都需要对方实例化后才能实现依赖注入。

假如类A先初始化：

  1. 通过Setter方法注入的话，会先调用空构造器，实例化类A；
  2. 随后容器想通过反射调用Setter方法来注入类B的实例，发现容器内并无类B的实例；
  3. 随后会立马对类B进行实例化，此时容器内已经存在类A的实例了，所以类B可以使用构造器方式注入，也可以使用Setter方式注入；
  4. 最后将实例化的类B注入到类A的实例中。

循环依赖解决了。

源码逻辑

接下来从源码角度分析循环依赖是如何解决的：

这里得到的Object对象就是想要创建的对象。进一步分析getSingleton方法：

@Nullable
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        synchronized (this.singletonObjects) {
            singletonObject = this.earlySingletonObjects.get(beanName);
            if (singletonObject == null && allowEarlyReference) {
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject();
                    this.earlySingletonObjects.put(beanName, singletonObject);
                    this.singletonFactories.remove(beanName);
                }
            }
        }
    }
    return singletonObject;
}

这个方法是Spring解决循环依赖的关键方法，在这个方法中，使用了三层列表来查询的方式，这三层列表分别是：

  • singletonObjects
  • earlySingletonObjects
  • singletonFactories

这个方法中用到的几个判断逻辑，体现了Spring解决循环依赖的思路，不过实际上对象被放入这三层的顺序是和方法查询的循序相反的，也就是说，在循环依赖出现时，对象往往会先进入singletonFactories，然后earlySingletonObjects，然后singletonObjects。

SingletonFactories中存放已经构造完成，但是还未注入属性的实例

下面看一下getSingleton的逻辑：

  1. 1

    Object singletonObject = this.singletonObjects.get(beanName);

    方法先从singletonObjects中获取对象，当Spring准备新建一个对象时，singletonObjects列表中是查询不到这个对象的。

  2. 1

    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName))

    除了判断null之外，有一个isSingletonCurrentlyInCreation的判断，实际上当Spring初始化了一个依赖注入的对象，但还没注入对象属性的时候，Spring会把这个bean加入singletonsCurrentlyInCreation这个set中，也就是把这个对象标记为正在创建的状态，这样，如果Spring发现要创建的bean在singletonObjects中没有，但在singletonsCurrentlyInCreation中有，基本上就可以认定为循环依赖了。

    1

    singletonObject = this.earlySingletonObjects.get(beanName);

  3. 1

    singletonObject = this.earlySingletonObjects.get(beanName);

    继续查询earlySingletonObjects列表，这是个为了循环依赖而存在的列表，从名字就可以看到，是个预创建的对象列表，刚刚创建的对象在这个列表里一般也没有。

  4. 1
2

    if (singletonObject == null && allowEarlyReference) {
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);

    earlySingletonObjects也没有则从singletonFactories中获取，前面说到singletonFactories是对象保存的第一步，实际上对象初始化后，可能还没有注入对象的依赖，就把对象放入了这个列表。如果是循环依赖，此时的singletonFactories中一般是会存在目标对象的，即解决方案中描述的，构造器已经被调用过了，但是属性还未注入。

    IoC容器把这个对象从singletonFactories移动到了earlySingletonObjects中。

  6. 其实上面5步已经执行完了该方法的代码，这里加的第6步是为了解释循环依赖的结果。在这个方法的代码之后，会把bean完整的进行初始化和依赖的注入，在完成了bean的初始化后，后面代码逻辑中会调用一个这样的方法：

    getSingleton(String beanName, ObjectFactory<?> singletonFactory)

    这个方法中有个小小的子方法addSingleton()，他的代码是这样的：

    protected void addSingleton(String beanName, Object singletonObject) {
    synchronized (this.singletonObjects) {
        this.singletonObjects.put(beanName, singletonObject);
        this.singletonFactories.remove(beanName);
        this.earlySingletonObjects.remove(beanName);
        this.registeredSingletons.add(beanName);
    }
}

    这个方法处理的是已经注入完依赖的bean，把bean放入singletonObjects中，并把bean从earlySingletonObjects和singletonFactories中删除，这个方法和上面分析的方法组成了Spring处理循环依赖的逻辑。
